# Engine API SSZ transport plan (2026-03-04)

## Scope confirmed
- Target branch: `unstable`
- SSZ content type: `application/octet-stream`
- Default behavior: JSON-RPC baseline + prefer SSZ only for endpoints mutually advertised via `engine_exchangeCapabilities`
- Fallback trigger: if SSZ endpoint not mutually advertised, stay on JSON-RPC for that method (no optimistic blind probe required)
- Coverage: engine methods currently used by Lodestar
- Integration layer: `@lodestar/api` first (avoid beacon-node-only ad hoc implementation)

## Spec surface (execution-apis PR #764)
Reference files:
- `src/engine/common.md` (Binary SSZ transport + negotiation/fallback)
- `src/engine/ssz-encoding.md` (versioned endpoints and request/response SSZ containers)

Endpoint families relevant to Lodestar engine client:
- payload submission: `POST /engine/v{N}/payloads`
- forkchoice update: `POST /engine/v{N}/forkchoice`
- payload retrieval: `GET /engine/v{N}/payloads/{payload_id}`
- payload bodies: `POST /engine/v{N}/payloads/bodies/by-hash|by-range`
- blobs: `POST /engine/v{N}/blobs`
- client version/capabilities: `POST /engine/v1/client/version`, `POST /engine/v1/capabilities`

## Implementation sketch
1. Add SSZ transport mode in `@lodestar/api` engine client with per-method endpoint mapping.
2. Add capability negotiation state from JSON-RPC `engine_exchangeCapabilities` and cache mutually supported SSZ endpoints.
3. Implement transport selection behavior:
   - for mutually advertised SSZ endpoints => use SSZ REST (`application/octet-stream`)
   - otherwise => use JSON-RPC path for that method
4. Wire beacon-node execution engine HTTP wrapper to use `@lodestar/api` transport selection.
5. Add tests:
   - SSZ happy path for mutually advertised endpoints (representative methods)
   - capability negotiation parsing/selection logic
   - JSON-RPC path for non-advertised endpoints
