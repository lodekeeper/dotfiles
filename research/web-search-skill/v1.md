## Executive Summary

We researched building a comprehensive web search skill that goes beyond our current Brave Search integration to find answers to arbitrary questions on the internet. The research surveyed 14+ search engine APIs, 6 self-hosted solutions, 20+ specialized sources, and produced a complete skill architecture.

**Key finding:** A multi-source search orchestrator with 4 providers in MVP (Brave + DuckDuckGo + GitHub Code Search + HN Algolia) can dramatically improve our search capability at zero additional cost. SearxNG self-hosted is the recommended "unlock" for unlimited free general web search. The total skill consists of ~800 lines of Python across 10 modules.

**Recommended approach:**
1. **Immediate (MVP):** Build `skills/web-search/` with query classification → parallel multi-source search → RRF ranking → optional LLM synthesis. Start with 4 free providers.
2. **Week 2:** Add specialized sources (Semantic Scholar, Wikipedia, Stack Exchange, ethresear.ch) and LLM synthesis with citations.
3. **Week 3+:** Deploy SearxNG, add learning/stats system, budget tracking.

**Cost:** $0/month for MVP. $0-15/month at scale (only if we upgrade Brave to paid tier for higher QPS).

## Problem Statement

Our current search capability is limited to a single tool: `web_search` (Brave Search API). This has several limitations:
- **Rate limited:** 1 QPS on free tier, causing frequent 429 errors
- **Single index:** Only Brave's 30B-page index — misses content from specialized sources
- **No query routing:** Same API for code questions, academic papers, Ethereum research, and general web queries
- **No result synthesis:** Returns raw results, agent must process them manually
- **No fallback:** If Brave is down or rate-limited, search fails entirely

We need a skill that can find answers to **any** question on the internet by intelligently querying the right sources.

## Free vs Paid Tradeoff Analysis

### For Our Usage Level (50-200 searches/day)

**Tier 1 — Completely Free (no key):**
SearxNG self-hosted + DuckDuckGo covers general web. All specialized sources (HN, Wikipedia, Semantic Scholar, arXiv, ethresear.ch, Bluesky, npm, crates.io) are free with no API key. This alone handles ~80% of queries.

**Tier 2 — Free with API Key:**
Brave free tier (1.7K/mo), Tavily (1K/mo), Exa (1K/mo), GitHub PAT (10/min), Stack Exchange key (10K/day). Trivial signup, big quality jump. Handles remaining ~15%.

**Tier 3 — Paid (only if needed):**
Serper at $0.30-1/1K is cheapest Google-quality access. Brave paid at $3/1K fixes our 1 QPS rate limit. At our usage level, this is $5-15/month max.

### Budget Scenarios

| Budget | Configuration | Coverage |
|--------|--------------|----------|
| $0/mo | SearxNG + DDG + free specialized | ~85% |
| $0/mo | + Free-tier API keys (Brave, Tavily, Exa, GitHub, SE) | ~95% |
| $10/mo | + Brave paid (higher QPS) | ~98% |
| $15/mo | + Serper (Google fallback) | ~99% |

**Sweet spot: $0/month** with SearxNG self-hosted + free API keys covers our needs.

## Proposed Architecture

### Pipeline Overview

```
Agent Question
    │
    ▼
[1. Query Classification] — Rule-based (85%) + LLM fallback (15%)
    │ → domains: [general, code, ethereum, academic, ...]
    ▼
[2. Route to Providers] — Domain → optimal provider combo
    │ → max 3 providers in parallel per query
    ▼
[3. Parallel Search] — Promise.allSettled, 8s timeout per provider
    │ → graceful partial failure handling
    ▼
[4. Aggregate + Deduplicate] — URL normalization + trigram similarity
    │
    ▼
[5. Rank via RRF] — Reciprocal Rank Fusion + domain boosts
    │
    ▼
[6. Content Extract] — web_fetch top-3 URLs (optional, depth=deep)
    │
    ▼
[7. LLM Synthesis] — Answer with inline citations [1][2][3]
```

### Key Design Decisions

1. **Query classification is rule-based first** — regex patterns cover ~85% of queries, LLM fallback only for ambiguous ones. Fast (<200ms), cheap (no tokens for most queries).

2. **Additive domain classification** — a query like "how does PeerDAS work in Lodestar?" matches [ethereum, code, academic], triggering all three provider sets.

3. **Reciprocal Rank Fusion (RRF)** for cross-source ranking — standard technique that works without needing compatible score scales.

4. **Circuit breaker per provider** — 3 consecutive failures → 60s cooldown → probe → recover. Prevents cascading failures.

5. **File-based rate limiting** — token bucket per provider in JSON files. No daemon needed, survives restarts.

### Implementation Plan

**MVP (Week 1):** Core orchestrator + 4 providers (Brave, DDG, GitHub Code, HN Algolia) + rule-based classifier + RRF aggregation. No synthesis. Output: ranked JSON results.

**Phase 2 (Week 2):** Add LLM synthesis with citations + 4 more providers (Semantic Scholar, Wikipedia, Stack Exchange, ethresear.ch) + content extraction via web_fetch.

**Phase 3 (Week 3+):** Deploy SearxNG, add Exa/Tavily/Perplexity providers, implement provider-stats learning system, budget tracking with alerts.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| DDG unofficial library gets blocked | Medium | Low | SearxNG/Brave as fallback |
| SearxNG upstream engines change markup | Medium | Low | Multiple engines provide redundancy |
| Brave free tier is insufficient at scale | Low | Medium | Upgrade to paid ($10/mo) or use SearxNG |
| Classification routes to wrong providers | Medium | Low | General fallback always included |
| LLM synthesis hallucinates beyond sources | Low | High | Strict prompt + citation requirement |
| Provider API deprecation | Low | Medium | Modular design — swap providers easily |

## Open Questions

1. **SearxNG deployment:** Should we deploy it now (Docker on this server, ~2GB RAM) or defer to Phase 3? Recommend: defer unless we hit Brave rate limits.

2. **Perplexity integration:** Returns AI answers, not raw results. Should it bypass the pipeline? Could be a fast "shortcut" for simple factual queries.

3. **Synthesis always-on vs opt-in:** For deep queries, synthesis is valuable. For simple lookups, it wastes tokens. Recommend: on for `depth=deep`, off for `depth=shallow`.

4. **Provider stats learning:** Track which sources perform best per domain over time. Recommend: defer to Phase 3, not needed for MVP.
